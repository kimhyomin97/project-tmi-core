## 1. 시스템 아키텍처

📊 **[시스템 구성도 (HTML)](./assets/diagrams/system-overview.html)**

| 컴포넌트 | 책임 | 위치 |
|---|---|---|
| Web Client | 사용자 입력 화면 | `project-tmi-web` |
| **TMI Core (이 레포)** | **HTTP API · 비즈니스 로직 · DB 접근 · 유사도 서버 호출** | `project-tmi-core` |
| TMI Similarity | 문장 임베딩, 의미 유사도 계산 | `project-tmi-similarity` |
| PostgreSQL | 문장·답안 이력 저장 | 인프라 |

### 1.1 멀티 서비스 분리 기준

언어 적합성을 기준으로 책임 분리.

| 책임 | 언어 / 런타임 | 이유 |
|---|---|---|
| HTTP API · DB · 비즈니스 로직 | Java / Spring Boot | 도메인 표현력, JPA 생산성, 운영 경험 |
| 문장 임베딩 · 유사도 계산 | Python / FastAPI | `sentence-transformers`·`numpy` 등 ML 생태계 우위 |

→ 사용자 단위가 아닌 **언어·라이브러리 단위**로 책임 분리.

### 1.2 통신 방식

| 항목 | 선택 | 이유 |
|---|---|---|
| 프로토콜 | HTTP/JSON | 단순, 디버깅 쉬움 |
| 호출 모델 | 동기 (request/response) | 단일 사용자, 즉시 응답 필요. 메시지 큐는 오버엔지니어링 |
| 타임아웃 | 2초 | 사용자 체감 한계 (3초 룰) 안쪽 |

## 2. 아키텍처 스타일

### 2.1 레이어드 + DDD 핵심 유지

**레이어드**: Controller → Service → Repository → Domain.

**DDD에서 유지하는 것**:
- 엔티티에 행위 포함 (anemic model 회피).
- 값 객체 활용 (예: `Difficulty`에 단어 수 범위 내장).
- 서비스는 얇게, 도메인 로직은 엔티티 안으로.

### 2.2 헥사고날에서 전환

초기 헥사고날(Ports & Adapters)을 적용했다 레이어드로 전환.

| 헥사고날 | 본 프로젝트 적합성 |
|---|---|
| 단일 도메인 · 단일 인바운드(REST) · 단일 아웃바운드(JPA) | 포트/어댑터의 이점이 작음 |
| UseCase 인터페이스 + 구현체, 도메인↔엔티티 매퍼 | 파일 수 2~3배, 보일러플레이트 과다 |
| 멀티 어댑터(REST·CLI·gRPC 등) 확장에 유리 | 멀티 어댑터 계획 없음 |

→ 단일 도메인·단일 입출구 시점엔 레이어드가 비용 대비 효과 우수.

## 3. 패키지 구조

### 3.1 트리

```
com.tmi.core
├── CoreApplication                  ← @SpringBootApplication
│
├── sentence/                        ← 도메인: 영어 문장 출제·조회
│   ├── controller/
│   ├── service/
│   ├── repository/
│   ├── domain/                      ← 엔티티 + 값 객체
│   └── dto/
│       ├── request/
│       └── response/
│
├── answer/                         ← 도메인: 답안 채점 (MVP: 응답 조립까지, 이력 저장은 MVP 이후)
│   ├── controller/
│   ├── service/
│   ├── domain/
│   └── dto/
│       ├── request/
│       └── response/
│
├── infra/                           ← 외부 시스템 클라이언트
│   └── similarity/
│       ├── SimilarityClient
│       └── dto/
│
└── common/                          ← 횡단 관심사
    ├── config/                      ← Spring 설정
    ├── define/                      ← 상수, 공용 enum
    └── exception/                   ← BaseException, GlobalHandler
```

### 3.2 분할 규칙

**도메인 우선 → 레이어 후속**: 먼저 도메인(`sentence/`, `answer/`)으로 나누고, 각 도메인 내부를 레이어(`controller/`, `service/`, …)로 다시 나눔.

**의존 방향**: `controller → service → repository → domain`. `domain`은 다른 패키지에 의존하지 않음.

**외부 시스템 클라이언트**: 도메인 패키지 안이 아닌 `infra/{시스템명}/` 아래. 도메인은 외부 통신 세부에 묶이지 않음.

> 패키지 명명·구조의 "규칙"은 [05. 개발 컨벤션 §1](./05_개발컨벤션.md). 본 §3은 "현재 인스턴스".

## 4. 핵심 처리 흐름

학습 사이클의 두 핵심 API — **문제 출제**와 **답안 채점** — 각각의 내부 흐름.

### 4.1 문제 출제

📊 **[시퀀스 다이어그램 (HTML)](./assets/diagrams/sequence-quiz-fetch.html)**

요청: `GET /api/v1/sentences/next?level=3&category=DAILY`

| # | 단계 | 비고 |
|---|---|---|
| 1 | `SentenceController` 수신 | 파라미터 검증 (Bean Validation) |
| 2 | `SentenceService.pickRandom(filter)` 호출 | |
| 3 | `SentenceRepository.findRandomBy(filter)` 호출 | DB의 `RANDOM()` 기반 1건 추출 |
| 4 | `SentenceEntity` 반환 | |
| 5 | 모범 답안(`koreanRef`)을 제외한 응답 DTO 조립 | |
| 6 | 200 OK + JSON 응답 | |

핵심 결정:
- **랜덤 추출은 DB가 책임** — `ORDER BY RANDOM() LIMIT 1` (Postgres). 후보 페이로드 전체 로딩 없음.
- **모범 답안은 클라이언트로 보내지 않음** — 응답 DTO 단계에서 명시적으로 제외.

### 4.2 답안 채점

📊 **[시퀀스 다이어그램 (HTML)](./assets/diagrams/sequence-answer-submit.html)**

요청: `POST /api/v1/answers`, body `{ sentenceId, userAnswer }`

| # | 단계 | 비고 |
|---|---|---|
| 1 | `AnswerController` 수신 | body 검증 |
| 2 | `AnswerService.score(request)` 호출 | |
| 3 | `SentenceRepository.findById(sentenceId)` | 모범 답안 조회. 없으면 `SentenceNotFound` |
| 4 | `SimilarityClient.score(userAnswer, koreanRef)` | 외부 호출 (§5) |
| 5 | 응답 DTO 조립 — 점수, 점수 등급, 모범 답안, 사용자 답안 | 학습 도움 → [01. 기획 §3](./01_기획.md#3-핵심-사이클) |
| 6 | 200 OK + JSON 응답 | |

핵심 결정:
- **외부 호출 실패는 도메인 예외로 변환** — `SimilarityUnavailable` 등. raw HTTP 예외 전파 금지.
- **답안 이력 저장은 MVP 범위 밖** — [01. 기획 §4.2](./01_기획.md#42-mvp-이후-검토-사항) 검토 사항.
- **점수 → 등급 변환은 Service 책임** — 0.9 이상 우수 / 0.7~0.9 양호 / 0.5~0.7 보통 / 미만 미흡 (임계값은 사용 후 조정).

## 5. 외부 시스템 연동 — Similarity

### 5.1 클라이언트 구성

| 항목 | 내용 |
|---|---|
| 위치 | `com.tmi.core.infra.similarity.SimilarityClient` |
| HTTP 라이브러리 | `RestClient` (Spring) |
| 엔드포인트 | `POST {baseUrl}/similarity` (상세 → [03. API 명세 §5](./03_API명세.md)) |
| 타임아웃 | connect 1초 · read 2초 |
| 재시도 | 없음 (단일 사용자, 즉시 응답) |

### 5.2 도메인 예외 매핑

| 외부 예외 | TMI 예외 | HTTP |
|---|---|---|
| `ResourceAccessException` (네트워크) | `SimilarityUnavailable` | 503 |
| `HttpServerErrorException` (5xx) | `SimilarityUnavailable` | 503 |
| `HttpClientErrorException` (4xx) | `SimilarityRequestInvalid` | 502 |

→ 외부 응답 본문이 사용자에게 그대로 노출되지 않음.

## 6. 장애 대응

### 6.1 Similarity 서버 장애

- 2초 타임아웃 → `SimilarityUnavailable` 발생 → 503 응답.
- 응답 body: `{ error: { code: "TMI.SIMILARITY.UNAVAILABLE", message: ... } }`.
- 사용자 UI는 "채점 서버가 잠시 응답하지 않습니다. 다시 시도해 주세요." 안내.

### 6.2 DB 장애

- Spring Boot 기본 트랜잭션 매니저가 `DataAccessException` 발생 → 글로벌 핸들러가 500.
- 헬스체크에 DB 상태 포함 (§6.3).

### 6.3 헬스체크

`GET /api/v1/health` 응답에 의존 컴포넌트 상태 포함.

```json
{ "status": "UP", "db": "UP", "similarity": "UP" }
```

- `db`: 짧은 query (`SELECT 1`) 성공 여부.
- `similarity`: `/health` 엔드포인트(외부) 호출 성공 여부, 200ms 캐시.

### 6.4 회복성 패턴 — MVP 이후

Circuit Breaker · Retry · Bulkhead 같은 회복성 패턴은 MVP 범위 밖. 실사용에서 외부 호출 실패가 빈번해지는 신호가 잡히면 단계적 도입 검토.
