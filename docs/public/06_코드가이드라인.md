## 1. Entity

### 1.1 핵심 결정

- `BaseEntity` 상속 — `created_at`/`updated_at`/`deleted_at` 자동 상속 (→ 99 §2.1)
- `@Entity`, `@Table(name = "...")` 단수형
- `@Getter` + `@NoArgsConstructor(access = AccessLevel.PROTECTED)` — setter 노출 X
- PK: `String id` UUID — 서비스 레이어에서 `UUID.randomUUID().toString()` 으로 생성
- enum 필드: `@Enumerated(EnumType.STRING) + @Column(length = N)`
- 생성·수정: 정적 팩토리 + 의미 메서드 — `create(...)` / `update{Field}(...)`
- Lombok `@Data` 금지, `@ToString` 비권장 — setter 자동 생성·`equals`/`hashCode` 모든 필드 기반이라 lazy 로딩·무한 루프 위험

### 1.2 코드 예시

```java
@Entity
@Table(name = "sentence")
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class SentenceEntity extends BaseEntity {

    @Id
    @Column(name = "id", length = 36)
    private String id;

    @Column(name = "english_text", nullable = false, columnDefinition = "TEXT")
    private String englishText;

    @Enumerated(EnumType.STRING)
    @Column(name = "level", nullable = false, length = 20)
    private Level level;

    public static SentenceEntity create(String englishText, Level level) {
        SentenceEntity entity = new SentenceEntity();
        entity.id = UUID.randomUUID().toString();
        entity.englishText = englishText;
        entity.level = level;
        return entity;
    }

    public void updateLevel(Level newLevel) {
        this.level = newLevel;
    }
}
```

### 1.3 `ddl-auto = validate` 스키마 동기화

`application.properties` 의 `spring.jpa.hibernate.ddl-auto=validate` — Hibernate 가 DDL 을 만들지 않음.

신규/변경 시 동기화 대상 **3 곳**:
1. `docs/public/04_데이터모델.md` — SSOT
2. `src/main/resources/init.sql` (또는 운영 DDL)
3. `*Entity.java`

`create` / `create-drop` / `update` 사용 금지 — Entity 와 DB 가 의도 없이 어긋남.

## 2. Repository

### 2.1 핵심 결정

- `JpaRepository<E, String>` 상속
- 메서드명 명명 규칙 활용
- soft delete 적용 시 모든 조회 메서드에 `AndDeletedAtIsNull` 명시
- 복잡 쿼리는 `@Query` (JPQL 우선, 필요 시 Native SQL)
- 반환 타입: 단건 `Optional<E>`, 다건 `List<E>` / `Page<E>`

### 2.2 코드 예시

```java
public interface SentenceRepository extends JpaRepository<SentenceEntity, String> {

    Optional<SentenceEntity> findByIdAndDeletedAtIsNull(String id);

    @Query(value = """
            SELECT * FROM sentence
            WHERE level = :level
              AND category = :category
              AND deleted_at IS NULL
            ORDER BY RANDOM()
            LIMIT 1
            """, nativeQuery = true)
    Optional<SentenceEntity> findRandomBy(@Param("level") Level level,
                                          @Param("category") Category category);
}
```

### 2.3 주의

- `findById` 기본 메서드는 soft delete 무시 — `AndDeletedAtIsNull` 명시 메서드 사용.
- enum 파라미터는 `@Enumerated(EnumType.STRING)` 컨버터가 자동 바인딩. SpEL `:#{#level.name()}` 트릭 불요.
- Native SQL 은 PostgreSQL 종속 — 의식적 선택.

## 3. Service

### 3.1 핵심 결정

- `@Service` + `@RequiredArgsConstructor` + `@Slf4j`
- 메서드별 `@Transactional` (class-level X)
- 조회 전용: `@Transactional(readOnly = true)`
- import: `org.springframework.transaction.annotation.Transactional` (jakarta 아님)
- dirty checking 의식적 미사용 — 변경 후 명시 `save()`
- 도메인 예외 throw → GlobalExceptionHandler 위임

### 3.2 코드 예시

```java
@Slf4j
@Service
@RequiredArgsConstructor
public class SentenceService {

    private final SentenceRepository sentenceRepository;

    @Transactional(readOnly = true)
    public SentenceEntity getRandom(Level level, Category category) {
        log.info("문장 출제: level={} category={}", level, category);
        return sentenceRepository.findRandomBy(level, category)
                .orElseThrow(() -> new SentenceException(
                        ESentenceErrorType.NOT_FOUND,
                        "level=" + level + ", category=" + category));
    }

    @Transactional
    public void updateText(String id, String newText) {
        SentenceEntity entity = sentenceRepository.findByIdAndDeletedAtIsNull(id)
                .orElseThrow(() -> new SentenceException(ESentenceErrorType.NOT_FOUND, id));
        entity.updateEnglishText(newText);
        sentenceRepository.save(entity);
    }
}
```

### 3.3 외부 호출은 트랜잭션 밖에서

외부 HTTP 호출(예: Similarity)을 `@Transactional` 블록 안에서 부르지 말 것 — DB 커넥션이 외부 응답 대기 시간 동안 점유됨.

```java
// 안티 패턴 — 외부 호출이 트랜잭션 안
@Transactional
public AnswerResult score(...) {
    SentenceEntity sentence = sentenceRepository.findById(...);
    SimilarityScore score = similarityClient.score(...);   // ← DB 커넥션 점유 중
    ...
}

// 권장 — 트랜잭션 분리
public AnswerResult score(...) {
    SentenceEntity sentence = loadSentence(...);    // 트랜잭션 안
    SimilarityScore score = similarityClient.score(...);   // 트랜잭션 밖
    return AnswerResult.from(sentence, score);
}

@Transactional(readOnly = true)
public SentenceEntity loadSentence(String id) { ... }
```

### 3.4 self-invocation 함정

같은 클래스 안의 `@Transactional` 메서드를 `this.X()` 로 호출하면 **프록시가 우회되어 트랜잭션·readOnly 가 적용되지 않음**. 분리하려면 다른 빈으로.

### 3.5 로깅 지점

| 위치 | 레벨 | 형태 |
|---|---|---|
| 주요 유스케이스 메서드 진입 | INFO | `log.info("문제 출제: level={} category={}", ...)` |
| 외부 호출 직전·직후 | INFO | `log.info("Similarity 호출"); log.info("Similarity 응답: score={}", ...)` |
| 외부 호출 실패 | ERROR | `log.error("Similarity 실패", e)` (Throwable 마지막 인자) |
| 도메인 예외 throw 직전 | (찍지 않음) | GlobalExceptionHandler 가 단일 로깅 |

## 4. Controller

### 4.1 핵심 결정

- `@RestController` + `@RequiredArgsConstructor`
- URL prefix 는 `application.properties` 의 `rest.api.version` 으로 통합 (→ 05 §4.1). 하드코딩 금지.
- 응답 항상 `ResponseEntity<T>`
- 요청 body 검증: `@Valid`
- 상태 코드 명시 — 200 / 201 + Location / 204
- 비즈니스 로직 X — Service 위임
- 예외 처리 X — GlobalExceptionHandler 위임

### 4.2 URL 버저닝

```properties
# application.properties
rest.api.version=/api/v1
```

```java
@RequestMapping("${rest.api.version}/sentences")
```

### 4.3 코드 예시 — GET

```java
@RestController
@RequestMapping("${rest.api.version}/sentences")
@RequiredArgsConstructor
public class SentenceController {

    private final SentenceService sentenceService;

    @GetMapping("/next")
    public ResponseEntity<SentenceResponse> getNext(@RequestParam Level level,
                                                    @RequestParam Category category) {
        SentenceEntity sentence = sentenceService.getRandom(level, category);
        return ResponseEntity.ok(SentenceResponse.from(sentence));
    }
}
```

### 4.4 코드 예시 — POST + 201

```java
@PostMapping
public ResponseEntity<SentenceResponse> create(@Valid @RequestBody CreateSentenceRequest request) {
    SentenceEntity sentence = sentenceService.create(request);
    return ResponseEntity
            .created(URI.create("/api/v1/sentences/" + sentence.getId()))
            .body(SentenceResponse.from(sentence));
}
```

### 4.5 Validation 핸들러 매핑

| 예외 | 발생 시점 | 응답 코드 |
|---|---|---|
| `MethodArgumentNotValidException` | `@Valid` body 검증 실패 | 400 `TMI.COMMON.INVALID_REQUEST` |
| `MethodArgumentTypeMismatchException` | `@RequestParam` 타입 변환 실패 (enum 등) | 400 `TMI.COMMON.INVALID_REQUEST` |
| `MissingServletRequestParameterException` | 필수 파라미터 누락 | 400 `TMI.COMMON.MISSING_PARAMETER` |
| `NoResourceFoundException` | 잘못된 URL | 404 `TMI.COMMON.NOT_FOUND_URL` |
| `HttpRequestMethodNotSupportedException` | 잘못된 HTTP 메서드 | 405 `TMI.COMMON.METHOD_NOT_SUPPORTED` |

모두 GlobalExceptionHandler 가 처리 (→ 99 §2.7). Controller 에서 `try/catch` 금지.

### 4.6 헬스체크

`GET /api/v1/health` (→ 03 §4) — Spring Actuator 재노출 X, 별도 `HealthController` 직접 작성.

- 위치: `common/controller/HealthController.java` (도메인 아님)
- 응답: `{ status, db, similarity }` (02 §6.3)

## 5. DTO

### 5.1 핵심 결정

- `record` + 중첩 `record`
- Request 는 Bean Validation (`@NotBlank`, `@Size` 등)
- Response 는 `from(Entity)` 정적 팩토리
- 위치: 도메인 패키지의 `dto/` (request·response 통합)

### 5.2 코드 예시 — Response

```java
public record SentenceResponse(
        String sentenceId,
        String englishText,
        Level level,
        Category category
) {
    public static SentenceResponse from(SentenceEntity sentence) {
        return new SentenceResponse(
                sentence.getId(),
                sentence.getEnglishText(),
                sentence.getLevel(),
                sentence.getCategory()
        );
    }
}
```

### 5.3 코드 예시 — Request

```java
public record SubmitAnswerRequest(
        @NotBlank @Size(max = 36) String sentenceId,
        @NotBlank @Size(min = 1, max = 500) String userAnswer
) {
}
```

### 5.4 enum 검증

- 필드가 enum 타입이면 Spring 이 자동 변환 — 잘못된 값은 `MethodArgumentTypeMismatchException` → 400.
- `@Pattern` 은 `CharSequence` 만 적용. 필드가 enum 이면 `@Pattern` 사용 X (무효).
- enum 으로 받지 않고 String 으로 받아 별도 검증할 때만 `@Pattern(regexp = "DAILY|BUSINESS|TECH")`.

### 5.5 Bean Validation 메시지

- 검증 실패 시 GlobalExceptionHandler 가 `MethodArgumentNotValidException` → `TMI.COMMON.INVALID_REQUEST`.
- 필드별 메시지가 필요하면 `@NotBlank(message = "{TMI.COMMON.INVALID_REQUEST.userAnswer}")` 형태로 messages.properties 키 사용.

## 6. Exception

### 6.1 핵심 결정

- 도메인별로 **1 예외 클래스** + **1 ErrorType enum** (→ 05 §5.3~§5.4)
- 정적 중첩 클래스 패턴 X
- ErrorType enum 이 HttpStatus 까지 보유 (`IErrorType.getStatus()`)
- 위치: `{domain}/exception/{Domain}Exception.java` + `{domain}/exception/E{Domain}ErrorType.java`
- 케이스 추가 시: enum 상수 1 줄 + messages.properties 1 줄 (예외 클래스는 그대로)

### 6.2 코드 예시 — 도메인 ErrorType enum

`sentence/exception/ESentenceErrorType.java`
```java
@Getter
@RequiredArgsConstructor
public enum ESentenceErrorType implements IErrorType {

    NOT_FOUND(HttpStatus.NOT_FOUND, "TMI.SENTENCE.NOT_FOUND"),
    DUPLICATE(HttpStatus.CONFLICT,  "TMI.SENTENCE.DUPLICATE"),
    ;

    private final HttpStatus status;
    private final String errorCode;
}
```

### 6.3 코드 예시 — 도메인 예외 클래스

`sentence/exception/SentenceException.java`
```java
public class SentenceException extends BaseException {

    public SentenceException(IErrorType errorType, Object... args) {
        super(errorType, args);
    }

    public SentenceException(IErrorType errorType, Throwable cause, Object... args) {
        super(errorType, cause, args);
    }
}
```

### 6.4 호출

```java
throw new SentenceException(ESentenceErrorType.NOT_FOUND, sentenceId);
```

### 6.5 주의

- 외부 시스템 (`infra/`) 예외도 같은 패턴 — `SimilarityException` + `ESimilarityErrorType` (→ 05 §5.7).
- 예외 throw 직전에 로그 찍지 말 것 — GlobalExceptionHandler 가 단일 로깅 (3.5 참조).
- 도메인 예외 추가 시 GlobalExceptionHandler 코드 변경 불요 — `BaseException` 단일 핸들러로 흡수.

## 7. Test

### 7.1 Phase 별 적용 범위

| Phase | 적용 |
|---|---|
| 4 (Sentence) | `@SpringBootTest` 통합 테스트 1~2 개 |
| 5 (Similarity 도입 후) | WireMock 추가 (외부 호출 mock) |
| 6 | 커버리지 보강 + 회귀 테스트 정비 |

### 7.2 핵심 결정

- 통합 테스트 위주
- 골격: `@SpringBootTest + @AutoConfigureMockMvc + @ActiveProfiles("test") + @Transactional`
- 메서드명 `shouldXxx_whenYyy`, `@DisplayName` 한국어
- DB: PostgreSQL 종속 기능(Native `RANDOM()`, jsonb)이 있어 **Testcontainers 권장** (H2 비추천)

### 7.3 코드 예시

```java
@SpringBootTest
@AutoConfigureMockMvc
@ActiveProfiles("test")
@Transactional
class SentenceControllerIntegrationTest {

    @Autowired private MockMvc mockMvc;
    @Autowired private SentenceRepository sentenceRepository;

    @Test
    @DisplayName("문장 출제 — 조건에 맞는 문장이 있으면 200")
    void shouldReturnSentence_whenMatchFound() throws Exception {
        sentenceRepository.save(SentenceEntity.create("Sounds good.", Level.LEVEL_1));

        mockMvc.perform(get("/api/v1/sentences/next")
                        .param("level", "LEVEL_1")
                        .param("category", "DAILY"))
                .andExpect(status().isOk())
                .andExpect(jsonPath("$.englishText").value("Sounds good."));
    }

    @Test
    @DisplayName("문장 출제 — 조건에 맞는 문장이 없으면 404")
    void shouldReturn404_whenNotFound() throws Exception {
        mockMvc.perform(get("/api/v1/sentences/next")
                        .param("level", "LEVEL_5")
                        .param("category", "TECH"))
                .andExpect(status().isNotFound())
                .andExpect(jsonPath("$.error.code").value("TMI.SENTENCE.NOT_FOUND"));
    }
}
```

### 7.4 주의

- `@Transactional` 테스트 롤백은 같은 트랜잭션 안에서만 — `@Async` / `Propagation.REQUIRES_NEW` 결합 시 안 됨.
- MockMvc 는 서블릿 컨테이너를 띄우지 않음. 필터·CORS 검증이 필요하면 `TestRestTemplate + @SpringBootTest(webEnvironment = RANDOM_PORT)`.
