# 06. 코드 가이드라인

> 각 계층의 클래스를 작성할 때 따를 코드 패턴 모음.

**대상 독자**: 이 프로젝트에 코드를 작성하는 사람  
**범위**: 계층별 작성 패턴 + 코드 예시  
**관련 문서**: [05. 개발 컨벤션](./05_개발컨벤션.md) — 컨벤션 규칙(네이밍, DB 명명, URL, 에러 코드 등). 본 문서는 그 규칙을 따르는 **코드 작성 예시**.

## 1. Entity

### 1.1 핵심 결정

- `BaseEntity` 상속 (`created_at`/`updated_at`/`deleted_at` 공통 컬럼 자동 상속)
- `@Entity`, `@Table(name = "...")` 단수형
- `@Getter` + `@NoArgsConstructor(access = AccessLevel.PROTECTED)` — setter 노출 X
- `@DynamicUpdate` (변경 컬럼만 UPDATE 쿼리에 포함)
- PK: `String id` UUID — 서비스 레이어에서 `UUID.randomUUID().toString()` 으로 생성
- enum 필드: `@Enumerated(EnumType.STRING) + @Column(length = N)`
- 생성·수정은 **정적 팩토리 + 의미 메서드** — `create(...)` / `update{Field}(...)`

### 1.2 코드 예시

```java
@Entity
@Table(name = "sentence")
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
@DynamicUpdate
public class SentenceEntity extends BaseEntity {

    @Id
    @Column(name = "id", length = 36)
    private String id;

    @Column(name = "english_text", nullable = false, columnDefinition = "TEXT")
    private String englishText;

    @Column(name = "korean_ref", nullable = false, columnDefinition = "TEXT")
    private String koreanRef;

    @Enumerated(EnumType.STRING)
    @Column(name = "level", nullable = false, length = 20)
    private Level level;

    @Enumerated(EnumType.STRING)
    @Column(name = "category", nullable = false, length = 20)
    private Category category;

    public static SentenceEntity create(String englishText, String koreanRef,
                                        Level level, Category category) {
        SentenceEntity entity = new SentenceEntity();
        entity.id = UUID.randomUUID().toString();
        entity.englishText = englishText;
        entity.koreanRef = koreanRef;
        entity.level = level;
        entity.category = category;
        return entity;
    }

    public void updateKoreanRef(String newRef) {
        this.koreanRef = newRef;
    }
}
```

### 1.3 주의

- **setter 노출 금지** — 도메인 의도 있는 update 메서드만 공개.
- 생성자 `PROTECTED` — JPA 만 호출 가능, 외부에서는 `create(...)` 만.
- enum 값(Java) ↔ DB 컬럼 값 ↔ API 응답 값 **3 자 일치** (예: `LEVEL_3`).

## 2. Repository

### 2.1 핵심 결정

- `JpaRepository<E, String>` 상속
- 메서드명 명명 규칙 활용 (`findByXxx`, `countByXxx`)
- **soft delete 적용 시 모든 조회 메서드에 `AndDeletedAtIsNull`** 명시
- 복잡 쿼리는 `@Query` + Native SQL 또는 JPQL
- 반환 타입: 단건은 `Optional<E>`, 다건은 `List<E>`

### 2.2 코드 예시

```java
public interface SentenceRepository extends JpaRepository<SentenceEntity, String> {

    Optional<SentenceEntity> findByIdAndDeletedAtIsNull(String id);

    @Query(value = """
            SELECT * FROM sentence
            WHERE level = :#{#level.name()}
              AND category = :#{#category.name()}
              AND deleted_at IS NULL
            ORDER BY RANDOM()
            LIMIT 1
            """, nativeQuery = true)
    Optional<SentenceEntity> findRandomBy(@Param("level") Level level,
                                          @Param("category") Category category);
}
```

### 2.3 주의

- `findById` (기본) 는 soft delete 무시 — `deletedAt IS NULL` 명시 메서드 사용.
- Native SQL 은 DB 종속 — PostgreSQL 종속 의식적 선택.
- 메서드명에 쓰인 필드명은 Entity 필드명과 정확히 일치해야 함 (`Level` 필드명이 `level`).

## 3. Service

### 3.1 핵심 결정

- `@Service` + `@RequiredArgsConstructor` + `@Slf4j`
- **메서드별 `@Transactional`** — class-level 사용 X
- 조회 전용 메서드: `@Transactional(readOnly = true)`
- import: `org.springframework.transaction.annotation.Transactional` (jakarta 아님)
- **dirty checking 의식적 미사용** — 변경 후 명시 `save()`
- 도메인 예외 throw → GlobalExceptionHandler 가 응답 조립

### 3.2 코드 예시

```java
@Slf4j
@Service
@RequiredArgsConstructor
public class SentenceService {

    private final SentenceRepository sentenceRepository;

    @Transactional(readOnly = true)
    public SentenceEntity getRandom(Level level, Category category) {
        return sentenceRepository.findRandomBy(level, category)
                .orElseThrow(() -> new SentenceException.NotFound(
                        "level=" + level + ", category=" + category
                ));
    }

    @Transactional
    public void updateKoreanRef(String id, String newRef) {
        SentenceEntity entity = sentenceRepository.findByIdAndDeletedAtIsNull(id)
                .orElseThrow(() -> new SentenceException.NotFound(id));
        entity.updateKoreanRef(newRef);
        sentenceRepository.save(entity);   // 명시 save
    }
}
```

### 3.3 주의

- class-level `@Transactional` 안 쓰는 이유: 의도 모호. 메서드별로 명시.
- `jakarta.transaction.Transactional` 아님 — Spring 것 사용.
- 명시 `save()`: dirty checking 으로도 동작은 하지만, 변경 시점이 명확.
- Controller 에서 try/catch 안 함 — 예외는 GlobalExceptionHandler 에 위임.

## 4. Controller

### 4.1 핵심 결정

- `@RestController` + `@RequiredArgsConstructor` + `@RequestMapping("/api/v1/<plural>")`
- 응답 항상 `ResponseEntity<T>` 로 반환
- 요청 body 검증: `@Valid`
- 상태 코드: 200 (`ResponseEntity.ok`), 201 (`ResponseEntity.created(URI).body(...)`), 204 (`ResponseEntity.noContent().build()`)
- 비즈니스 로직 X — Service 위임
- 예외 처리 X — GlobalExceptionHandler 위임

### 4.2 코드 예시

```java
@RestController
@RequestMapping("/api/v1/sentences")
@RequiredArgsConstructor
public class SentenceController {

    private final SentenceService sentenceService;

    @GetMapping("/next")
    public ResponseEntity<SentenceResponse> getNext(@RequestParam Level level,
                                                    @RequestParam Category category) {
        SentenceEntity sentence = sentenceService.getRandom(level, category);
        return ResponseEntity.ok(SentenceResponse.from(sentence));
    }
}
```

### 4.3 주의

- enum 파라미터(`Level`, `Category`)는 Spring 이 문자열 → enum 자동 변환. 잘못된 값 → 400.
- Controller 가 받는 건 Service 결과 → DTO 변환. 비즈니스 결정은 Service.
- 응답 상태 코드를 본문에 따라 분기하지 말 것 — 일관성 ↓.

## 5. DTO

### 5.1 핵심 결정

- `record` + 중첩 `record`
- Request 는 Bean Validation (`@NotBlank`, `@Size`, `@Pattern` 등)
- Response 는 `from(Entity)` 정적 팩토리로 Entity → DTO 변환
- 위치: 도메인 패키지의 `dto/` (request·response 통합)

### 5.2 코드 예시 — Response

```java
public record SentenceResponse(
        String sentenceId,
        String englishText,
        Level level,
        Category category
) {
    public static SentenceResponse from(SentenceEntity sentence) {
        return new SentenceResponse(
                sentence.getId(),
                sentence.getEnglishText(),
                sentence.getLevel(),
                sentence.getCategory()
        );
    }
}
```

### 5.3 코드 예시 — Request

```java
public record SubmitAnswerRequest(
        @NotBlank @Size(max = 36) String sentenceId,
        @NotBlank @Size(min = 1, max = 500) String userAnswer
) {
}
```

### 5.4 주의

- record 는 불변 — Entity 가 변경되어도 DTO 는 그대로 (의도된 분리).
- enum 값 검증: `@RequestParam`/`@RequestBody` 에서 enum 타입 받으면 자동 변환. 직접 `@Pattern(regexp = "DAILY|BUSINESS|TECH")` 도 가능.
- Request·Response 같은 모양이라도 분리 (의도 명확화).

## 6. Exception

### 6.1 핵심 결정

- 도메인 묶음 외부 클래스 + 정적 중첩 클래스 (`SentenceException.NotFound`)
- 각 정적 중첩 클래스는 `BaseException` 직접 상속
- 외부 클래스는 인스턴스화 금지 (`private` 생성자)
- 케이스마다 `ETmiErrorType` enum 에 코드 추가 + `messages.properties` 에 메시지 추가
- HttpStatus 는 예외 생성 시 명시 (enum 에 묶지 않음)

### 6.2 코드 예시

```java
public class SentenceException {

    private SentenceException() {}

    /** 조건에 맞는 영어 문장이 없음 */
    public static class NotFound extends BaseException {
        public NotFound(String detail) {
            super(HttpStatus.NOT_FOUND, ETmiErrorType.SENTENCE_NOT_FOUND, detail);
        }
    }

    /** 같은 영어 문장이 이미 존재 */
    public static class Duplicate extends BaseException {
        public Duplicate(String englishText) {
            super(HttpStatus.CONFLICT, ETmiErrorType.SENTENCE_DUPLICATE, englishText);
        }
    }
}
```

### 6.3 호출

```java
throw new SentenceException.NotFound(id);
```

### 6.4 주의

- 새 케이스 추가 시: `ETmiErrorType` enum 상수 1 개 + `messages.properties` 한 줄 + 정적 중첩 클래스 1 개.
- `1 케이스 = 1 정적 중첩 클래스` — 의미 있는 예외만. 일회성 사소한 예외는 `BaseException` 익명 자식이나 `IllegalArgumentException` 직접 throw 도 OK (다만 권장 X).
- GlobalExceptionHandler 가 `BaseException` 단일 핸들러로 잡으므로 도메인 예외 추가 시 핸들러 코드 변경 불필요.

## 7. Test (Phase 6 시점에 구체화)

### 7.1 핵심 결정

- 통합 테스트 위주 — 단위 테스트는 적게
- `@SpringBootTest + @AutoConfigureMockMvc + @ActiveProfiles("test") + @Transactional` 4 어노테이션 골격
- 메서드명: `shouldXxx_whenYyy` 패턴
- `@DisplayName` 한국어 권장
- H2 또는 Testcontainers (PostgreSQL) 선택
- 외부 시스템 호출은 WireMock + `@DynamicPropertySource` 로 mock

### 7.2 코드 예시

```java
@SpringBootTest
@AutoConfigureMockMvc
@ActiveProfiles("test")
@Transactional
class SentenceControllerIntegrationTest {

    @Autowired
    private MockMvc mockMvc;

    @Autowired
    private SentenceRepository sentenceRepository;

    @Test
    @DisplayName("문장 출제 — 조건에 맞는 문장이 있으면 200 반환")
    void shouldReturnSentence_whenMatchFound() throws Exception {
        // given
        sentenceRepository.save(SentenceEntity.create(
                "Sounds good.", "좋아요.", Level.LEVEL_1, Category.DAILY));

        // when / then
        mockMvc.perform(get("/api/v1/sentences/next")
                        .param("level", "LEVEL_1")
                        .param("category", "DAILY"))
                .andExpect(status().isOk())
                .andExpect(jsonPath("$.englishText").value("Sounds good."));
    }

    @Test
    @DisplayName("문장 출제 — 조건에 맞는 문장이 없으면 404")
    void shouldReturn404_whenNotFound() throws Exception {
        mockMvc.perform(get("/api/v1/sentences/next")
                        .param("level", "LEVEL_5")
                        .param("category", "TECH"))
                .andExpect(status().isNotFound())
                .andExpect(jsonPath("$.error.code").value("TMI.SENTENCE.NOT_FOUND"));
    }
}
```

### 7.3 주의

- `@Transactional` 이 테스트 후 롤백 → 데이터 격리.
- H2 + PostgreSQL 함수(`RANDOM()`) 호환성 확인 필요 — 안 되면 Testcontainers 권장.
- WireMock 포트는 dynamic + `@DynamicPropertySource` 로 주입 (테스트 간 충돌 회피).
- 단위 테스트는 Mockito 로 Service 만 분리해 검증 가능 — 다만 통합으로 흡수하는 게 흔함.
