# 04. 데이터 모델

> TMI Core가 사용하는 데이터의 구조 — 테이블, 컬럼, 제약, 인덱스, 스키마 관리 방식.

**대상 독자**: 스키마를 추가·변경하거나 데이터를 직접 조회하는 개발자  
**범위**: 데이터의 형태와 스키마 관리 절차  
**관련 문서**: [05. 개발 컨벤션 §3 DB 스키마](./05_개발컨벤션.md#3-db-스키마) — 명명·타입 규칙

## 1. 개요

MVP는 **`sentence` 단일 테이블**.
답안 이력(`answer_history`)·사용자(`user`) 등은 MVP 이후 검토 사항 (→ [01 §4.2](./01_기획.md#42-mvp-이후-검토-사항)).

테이블이 늘면 ERD HTML 다이어그램을 본 섹션에 추가.

## 2. 테이블 명세

### 2.1 sentence — 영어 문장

학습에 사용되는 영어 문장과 모범 답안(한국어).

| 컬럼 | 타입 | NULL | 기본값 | 설명 |
|---|---|---|---|---|
| `id` | `varchar(36)` | NN | — | PK, 서비스 레이어에서 UUID 생성 |
| `english_text` | `text` | NN | — | 출제용 영어 원문 |
| `korean_ref` | `text` | NN | — | 모범 답안 (한국어) |
| `level` | `varchar(20)` | NN | — | 난이도 코드, `LEVEL_1` ~ `LEVEL_5` |
| `category` | `varchar(20)` | NN | — | 카테고리 코드, `DAILY` / `BUSINESS` / `TECH` |
| `created_at` | `timestamptz` | NN | `now()` | 생성 시각 |
| `updated_at` | `timestamptz` | NN | `now()` | 마지막 수정 시각 |
| `deleted_at` | `timestamptz` | NULL | `NULL` | soft delete, NULL이면 활성 |

**제약**

| 종류 | 정의 |
|---|---|
| PK | `id` |
| CHECK | `level IN ('LEVEL_1', 'LEVEL_2', 'LEVEL_3', 'LEVEL_4', 'LEVEL_5')` |
| CHECK | `category IN ('DAILY', 'BUSINESS', 'TECH')` |

**DDL**

```sql
CREATE TABLE sentence (
    id            varchar(36) PRIMARY KEY,
    english_text  text        NOT NULL,
    korean_ref    text        NOT NULL,
    level         varchar(20) NOT NULL
                  CHECK (level IN ('LEVEL_1', 'LEVEL_2', 'LEVEL_3', 'LEVEL_4', 'LEVEL_5')),
    category      varchar(20) NOT NULL
                  CHECK (category IN ('DAILY', 'BUSINESS', 'TECH')),
    created_at    timestamptz NOT NULL DEFAULT now(),
    updated_at    timestamptz NOT NULL DEFAULT now(),
    deleted_at    timestamptz NULL
);
```

### 2.2 향후 테이블 — MVP 이후

| 테이블 | 도메인 | 도입 시점 |
|---|---|---|
| `answer_history` | 답안 이력 (append-only) | 답안 이력 저장 기능 도입 시 |
| `user` | 사용자 | 멀티 사용자 전환 시 |

## 3. 인덱스

출제 쿼리(`WHERE level=? AND category=? AND deleted_at IS NULL ORDER BY RANDOM() LIMIT 1`)를 가속.

| 인덱스 | 컬럼 | 종류 | 용도 |
|---|---|---|---|
| `idx_sentence_filter` | `(level, category)` | partial (`WHERE deleted_at IS NULL`) | 출제 시 필터링 |

```sql
CREATE INDEX idx_sentence_filter
    ON sentence (level, category)
    WHERE deleted_at IS NULL;
```

> 부분 인덱스로 활성 행만 인덱싱 → 인덱스 크기·갱신 비용 절감.
> 출제 쿼리는 `ORDER BY RANDOM()` 특성상 정렬 비용은 남지만, 후보 집합 자체가 인덱스로 좁아짐.

## 4. 스키마 관리

### 4.1 도구 선택

**Flyway·Liquibase 등 마이그레이션 도구 미도입** (→ [memory: project-tmi-no-flyway]).

이유: 사이드 프로젝트 규모, 단일 사용자, 단일 배포 환경 → 자동 마이그레이션 비용 대비 이득 작음.

### 4.2 SSOT

스키마의 단일 진실 소스 두 곳, 항상 동기화 유지.

| SSOT | 형태 | 역할 |
|---|---|---|
| 본 문서 §2 | Markdown 표·DDL | 개념 정의, 문서 검토용 |
| `src/main/resources/db/schema.sql` | SQL 파일 | 실제 DB 적용용 |

### 4.3 변경 절차

```
① 본 문서 §2 갱신 (개념·DDL)
       ↓
② schema.sql 수정
       ↓
③ 로컬 DB 적용
       ↓
④ JPA Entity 코드 수정 (필드·타입 일치)
       ↓
⑤ 동작 확인 (ddl-auto=validate가 불일치 시 부팅 실패)
```

문서·SQL·코드 세 곳을 같은 순서로 갱신, 부팅 검증으로 동기화 확인.

### 4.4 적용 방식

| 환경 | 방식 |
|---|---|
| 로컬 (개발) | PostgreSQL 컨테이너 초기화 스크립트에 `schema.sql` 마운트, 또는 직접 `psql` 실행 |
| 운영 | → [06. 배포·운영](./06_배포·운영.md) (배포 대상 결정 후 절차 확정) |

### 4.5 JPA `ddl-auto` 정책

| 환경 | 값 | 이유 |
|---|---|---|
| 로컬·운영 | `validate` | 스키마 자동 변경 금지, Entity ↔ DB 불일치 시 부팅 실패 |
| 테스트 | `create-drop` | 테스트 간 격리, 빠른 초기화 |

## 5. 시드 데이터

### 5.1 대상

| 데이터 | 필요성 |
|---|---|
| 영어 문장 카탈로그 (`sentence`) | MVP 첫 사용에 최소 카테고리·난이도별 5건 이상 필요 |
| `level` / `category` 코드 | enum (`varchar + CHECK`)이라 별도 시드 불요 |

### 5.2 위치

`src/main/resources/db/seed.sql`

```sql
-- 예시
INSERT INTO sentence (id, english_text, korean_ref, level, category)
VALUES
  ('01HTM3...', 'Sounds good.',                          '좋아요.',                'LEVEL_1', 'DAILY'),
  ('01HTM4...', 'What did you do yesterday?',            '어제 뭐 했어요?',          'LEVEL_2', 'DAILY'),
  ('01HTM5...', 'I had a rough day at work today.',      '오늘 회사에서 고단한 하루를 보냈다.', 'LEVEL_3', 'DAILY'),
  ...
;
```

### 5.3 적용

- 로컬: `schema.sql` 적용 후 `seed.sql` 수동 실행 (`psql -f seed.sql`)
- 운영: 별 데이터 모집·관리 절차 (MVP 이후 결정)

> 콘텐츠 운영(문장 추가·수정 API)은 MVP 이후 검토 사항 (→ [01 §4.2](./01_기획.md#42-mvp-이후-검토-사항)).
